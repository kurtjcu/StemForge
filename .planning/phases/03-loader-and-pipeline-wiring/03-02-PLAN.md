<objective>
Wire drum stem routing into MidiPipeline.run() so drum-labeled stems are transcribed via ADTOF instead of BasicPitch.

Purpose: This is the core routing change that makes drum MIDI extraction work end-to-end. Without this, drum stems silently fall through to BasicPitch (pitched-instrument model), producing garbled output. After this plan, a user can upload a drum stem, click Extract MIDI, and receive correct GM channel-10 percussion MIDI.

Output: `_DRUM_STEM_LABELS` frozenset, third routing branch in `run()`, `is_drum=True` in `_build_stem_midi()`, 3-stage progress callbacks on drum path, post-loop drum model eviction. Test file proving all routing and integration behavior.

Note on VRAM eviction: `backend/api/midi.py` does NOT need modification for VRAM release. `AdtofBackend.evict()` (pipelines/adtof_backend.py:186-198) already calls `torch.cuda.empty_cache()` when the model was on a CUDA device. The eviction chain is: `MidiPipeline.run()` -> `self._loader.evict_drum_model()` -> `AdtofBackend.evict()` (which includes `model.cpu()` + `torch.cuda.empty_cache()`). This is called inside the pipeline run after all drum stems are processed, ensuring VRAM is freed before the next job.
</objective>

```python
class AdtofBackend:
    def evict(self) -> None:
        """Release model weights from GPU/CPU memory.
        Moves the model to CPU before clearing the reference so PyTorch can
        free GPU memory.  Calls torch.cuda.empty_cache() when the model
        was loaded on a CUDA device."""
        if self._model is not None:
            self._model.cpu()
            self._model = None
            if self._device.startswith("cuda"):
                torch.cuda.empty_cache()
```
